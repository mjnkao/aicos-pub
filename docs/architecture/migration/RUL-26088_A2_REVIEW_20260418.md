# RUL-26088 A2 Review

Date: 2026-04-18
Reviewer role: A2-R / A2-C for AICOS restructuring
Reviewed file: `docs/New design/RUL-26088_AICOS_Temporary_Role_Rules_For_Codex.md`

## Purpose

This note reviews whether `RUL-26088` can be applied effectively to the current
AICOS repo after the restructuring cleanup.

The source rule file has been copied into the repo at:

```text
docs/New design/RUL-26088_AICOS_Temporary_Role_Rules_For_Codex.md
```

## Summary

`RUL-26088` is directionally correct and useful. Its strongest contribution is
role clarity:

- A1 works on project/business reality.
- A2 works on the AICOS system reality.
- Codex should usually be treated as A2-C when changing AICOS code,
  structure, runtime, config, integrations, kernel, serving, or service skills.
- A2-R and A2-C should remain separate modes.

However, the rule file cannot be applied as-is because parts of its startup
reading order refer to the old repository layout that has now been moved into:

```text
backup/pre-restructure-20260418/
```

If applied literally, it would reintroduce the old problem: agents would start
by reading legacy folders and old context that were intentionally isolated.

## What Applies Well

### 1. A1/A2 Separation

This should be adopted.

The distinction is compatible with the new architecture:

- A1 lives mostly in `agent-repo/classes/a1-work-agents/`.
- A2 lives mostly in `agent-repo/classes/a2-service-agents/`.
- Human review lives in `agent-repo/classes/humans/`.
- Durable project reality lives in `brain/`.

This is especially important because Codex is currently changing AICOS itself,
so Codex should be treated as A2 by default for this work.

### 2. Codex Default For Current Work

This should be adopted.

When Codex is building or modifying:

- `packages/aicos-kernel/`
- `serving/`
- `backend/`
- `integrations/`
- `agent-repo/classes/a2-service-agents/skills/`
- repo structure
- CLI commands
- runtime/config/MCP lanes

Codex is acting as A2-C.

When Codex is reviewing policy, architecture, tradeoffs, or service design
without large edits, Codex is acting as A2-R.

### 3. A2 Boundary Guardrails

This should be adopted.

The guardrails match the current architecture:

- do not make `backend/` truth;
- do not make `agent-repo/` project truth;
- do not collapse `canonical/`, `working/`, and `evidence`;
- do not hardcode evolving service intelligence too early;
- do not overbuild the MVP;
- do not make GStack mandatory;
- preserve migration safety.

These rules are already aligned with the active build.

### 4. A2-R vs A2-C Mode Separation

This should be adopted.

It prevents Codex from making structural code changes while the policy is still
unclear. This is useful for the next phase, especially around:

- `aicos option choose`;
- full JSON Schema validation;
- GBrain import/index activation;
- MCP stdio wrapper design.

## What Does Not Apply As-Is

### 1. Startup Reading Order Is Too Heavy For Current Repo State

`RUL-26088` says agents should always read:

1. `RUL-26088`;
2. `ARC-26071`;
3. `PLN-26072`;
4. broader architecture docs;
5. current repo structure;
6. old lanes like `canonical/`, `brains/`, `gateway/`, `bridges/`, `reports/`,
   `scripts/`.

This was reasonable before the cleanup, but it is not safe as a literal startup
rule now.

The old lanes were moved into backup specifically so they would not be read by
default. Requiring agents to read them again would undo the cleanup.

### 2. References To Old Root Folders Are Stale

The rule file still names old root paths:

- `canonical/`
- `brains/`
- `gateway/`
- `bridges/`
- `reports/`
- broad old `scripts/`

Those paths are no longer active root lanes. They now live under:

```text
backup/pre-restructure-20260418/root/
backup/pre-restructure-20260418/scripts/
```

They should be treated as legacy reference material only.

### 3. "Always Read This File First" Conflicts With Current Restructuring Mode

The root `AGENTS.md` intentionally says there is no mandatory startup reading
list. This was done to prevent old context preloading from dominating current
work.

`RUL-26088` can be a role-classification reference, but making it mandatory for
every task may become too heavy and may conflict with the lightweight active
root rule.

A better rule is:

Read RUL-26088 when the task involves role ambiguity, A1/A2 lane decisions, or
changes to AICOS itself.

## Recommended Revision

`RUL-26088` should be adopted after a small revision.

Recommended changes:

### Change 1: Make RUL-26088 A Role Classifier, Not A Full Startup Protocol

Replace "always read this file first" with:

```text
For AICOS system changes, use RUL-26088 to classify the current role as A1,
A2-R, or A2-C before choosing work lanes.
```

### Change 2: Update Startup Reading Order For Current Root

For A2 work in the current repo, the reading order should be:

1. `AGENTS.md`
2. `docs/New design/RUL-26088_AICOS_Temporary_Role_Rules_For_Codex.md`
3. the specific design docs named by the current task;
4. current root structure;
5. relevant new-architecture lanes:
   - `brain/`
   - `agent-repo/`
   - `packages/aicos-kernel/`
   - `serving/`
   - `backend/`
   - `integrations/`
6. backup material only when explicitly needed.

### Change 3: Replace Old Root Lane References With Backup References

Where the file says to inspect:

```text
canonical/
brains/
gateway/
bridges/
reports/
scripts/
```

it should say:

```text
backup/pre-restructure-20260418/root/canonical/
backup/pre-restructure-20260418/root/brains/
backup/pre-restructure-20260418/root/gateway/
backup/pre-restructure-20260418/root/bridges/
backup/pre-restructure-20260418/root/reports/
backup/pre-restructure-20260418/scripts/
```

and only for explicit migration, provenance, or comparison tasks.

### Change 4: Add Current Active Root Paths

The revised file should name the current active paths:

```text
brain/
agent-repo/
packages/aicos-kernel/
serving/
backend/
integrations/
docs/New design/
docs/migration/
tools/gbrain/
```

### Change 5: Add A2 Default For The Current User Request

For the current phase:

```text
Codex is A2-C when it edits AICOS structure, code, scripts, configs, CLI,
integrations, service skills, backend, serving, or kernel.
```

This can be applied immediately.

## Practical Decision

Do not apply `RUL-26088` literally as the new root startup protocol.

Apply it as:

1. a role-classification rule;
2. an A1/A2 boundary rule;
3. an A2 safety rule;
4. a source for future permanent A1/A2 rule files.

Do not apply its old startup reading order until it is rewritten for the new
root structure and backup model.

## Recommended Permanent Placement Later

Split `RUL-26088` into permanent files:

```text
agent-repo/shared/shared-access-rules/role-classification-policy.md
agent-repo/classes/a1-work-agents/rules/lane-boundaries.md
agent-repo/classes/a2-service-agents/rules/lane-boundaries.md
agent-repo/classes/a2-service-agents/rules/reasoning-vs-coding-mode.md
agent-repo/classes/humans/role-policies/manager-review-boundary.md
```

This would fit the new architecture better than keeping one broad temporary
startup document forever.

## Operator Decision: Defer Splitting

Decision date: 2026-04-18

The proposed split is directionally accepted, but it should not be implemented
yet.

For now, keep `RUL-26088` as a temporary role-rules reference and surface the
important operating rule directly in `AGENTS.md` so current agents understand
the A1/A2 distinction.

Reason:

- the rules are still changing;
- the current architecture is still being stabilized;
- splitting too early could create many small permanent files that need churn;
- A1/A2 boundaries should be observed through more implementation/review cycles
  before being frozen into class-specific permanent files.

Recommended trigger for splitting:

1. the active root structure stays stable through at least one more build/review
   pass;
2. A1 and A2 tasks have both been exercised against the new `brain/`,
   `agent-repo/`, `serving/`, and `packages/aicos-kernel/` lanes;
3. the team agrees which parts of `RUL-26088` are stable operating rules versus
   temporary scaffolding;
4. `AGENTS.md` starts feeling too large or too mixed-purpose;
5. there is a concrete need for agents to load class-specific rules without
   reading the whole temporary file.

Until then, do not split `RUL-26088` into permanent files.

## Final Recommendation

Adopt the spirit of `RUL-26088` now, especially A1/A2 separation and Codex as
A2 for current AICOS system work.

Revise the startup-reading section before making it mandatory.

The current effective rule should be:

```text
Use RUL-26088 to classify role and preserve A1/A2 boundaries.
Use AGENTS.md and current task docs for active reading order.
Use backup only when explicitly needed.
```

Current operator preference:

```text
Keep the temporary role rules visible through AGENTS.md for now.
Defer splitting RUL-26088 until the architecture and role rules reach a more
stable point.
```
